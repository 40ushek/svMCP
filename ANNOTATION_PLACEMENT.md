# Annotation Placement — Стратегии и алгоритмы

Справочный файл: известные подходы к автоматической расстановке меток и размеров на технических чертежах.
Цель — не изобретать велосипед при дальнейшем развитии `resolve_mark_overlaps` и смежных инструментов.

---

## Задача

**Point-feature label placement**: для каждой точки (якорь = деталь/болт) разместить текстовый прямоугольник так, чтобы:
1. Метка была как можно ближе к якорю (короткая выноска)
2. Не перекрывала другие метки
3. Не перекрывала размеры, рамки видов, линии детали

Это NP-hard задача в общем случае, но для N < 100 решается хорошо.

---

## Классические алгоритмы для меток

### 1. Imhof (1975) — правила приоритета позиций
Эмпирические правила картографов, до сих пор используются как cost function:

| Приоритет | Позиция метки относительно точки |
|---|---|
| 1 | Справа сверху |
| 2 | Справа снизу |
| 3 | Слева сверху |
| 4 | Слева снизу |
| 5 | Прямо справа |
| 6 | Прямо слева |
| 7 | Прямо сверху |
| 8 | Прямо снизу |

Для каждой из 8 позиций генерируется кандидат. Выбирается лучший без перекрытий.

### 2. Christensen et al. (1995) — Simulated Annealing
Лучший практический результат для N < 500:
- Генерировать 8 кандидатов на метку
- Начальное размещение: greedy по приоритету Imhof
- SA: случайно менять позицию одной метки, принимать ухудшение с p = e^(-ΔE/T)
- T охлаждается по расписанию (обычно T *= 0.9 каждые N шагов)
- **Cost function**: Σ (overlap_area × k_overlap + dist_to_anchor × k_dist + priority_cost)

Референс: Christensen J. et al., "Cartographic Label Placement by Simulated Annealing" (1995)

### 3. Hirsch (1982) — Greedy с backtracking
Проще SA, хуже качество, но быстрее:
1. Отсортировать метки по числу конфликтующих кандидатов (самые "зажатые" — первыми)
2. Для каждой метки выбрать лучшего кандидата без перекрытий
3. Если нет свободного кандидата — сдвинуть уже размещённые метки

### 4. Force-directed / Spring simulation
Непрерывная оптимизация, естественный результат:
- Пружина к якорю: `F_spring = k_s × (anchor − pos)` — тянет метку к детали
- Отталкивание при перекрытии: `F_repulse = k_r × overlap_vector`
- Затухание скорости: `vel *= damping`
- Итерировать до `|F| < ε`

Параметры для технических чертежей (рекомендуемые):
- `k_spring = 0.3` (мягкая пружина, чтобы не тащить через препятствия)
- `k_repulse = 1.5` (жёсткое отталкивание, перекрытий быть не должно)
- `damping = 0.6`
- `dt = 0.5`

### 5. Текущая реализация в svMCP (`resolve_mark_overlaps`)
Итеративный AABB push-apart с anchor bias:
- Обе метки в паре двигаются (не только одна)
- Смещение делится 70/30: та что дальше от якоря двигается больше
- Якорь = `LeaderLinePlacing.StartPoint` в координатах листа: `sheetPt = viewOrigin + startPt / scale`
- Применяется только `InsertionPoint`; `StartPoint` не трогается

## Алгоритмы для размеров (Dimension Placement)

### Правила ISO 129 / ASME Y14.5

1. **Меньшие размеры — ближе к контуру детали**, большие — дальше
2. Размерные линии **не пересекаются** (если возможно)
3. Размеры располагаются **вне контура** детали
4. Параллельные размеры выравниваются (одинаковый `Distance` от контура, с шагом)

### Алгоритм упорядочивания цепочки размеров

Для набора размеров вдоль одной оси:
1. Вычислить длину каждого размера (расстояние между точками сегмента)
2. Отсортировать по длине (по возрастанию)
3. Назначить `Distance`: `D_i = D_min + i × step`
   - `D_min` ≈ 8–12 мм (зависит от масштаба)
   - `step` ≈ 7–10 мм

Это даёт классическую "лесенку" размеров, принятую в строительных чертежах.

### Предотвращение пересечений выносок

Если два размера имеют сегменты, чьи выноски пересекаются — поменять их `Distance` местами.
Это задача минимизации инверсий (аналог сортировки пузырьком по числу пересечений).

## Препятствия (obstacles)

Метки и размеры должны обходить:
- Другие метки (mark text boxes)
- Размерные линии (dimension lines)
- Рамки видов (view frames)
- Линии контура детали (доступны через `DrawingObject` геометрию, сложно)

В Tekla API:
- `Mark.GetAxisAlignedBoundingBox()` — доступен ✅
- `StraightDimensionSet.GetAxisAlignedBoundingBox()` — **недоступен** ❌
  → bbox нужно вычислять вручную из сегментов + `Distance`
- `View` bbox: `originX/Y + width/height` — доступен ✅

## Трансформация координат: view → sheet

`LeaderLinePlacing.StartPoint` и геометрия деталей находятся в координатах вида (view space).
`InsertionPoint` и bbox меток — в координатах листа (sheet space).

Для стандартных ортогональных видов:
```
sheetX = view.Origin.X + viewPoint.X / view.Attributes.Scale
sheetY = view.Origin.Y + viewPoint.Y / view.Attributes.Scale
```

Для произвольных видов (разрезы, изометрия) нужен `view.GetCoordinateSystem()` —
матрица поворота, проецирующая 3D model coords в 2D view coords.

## COG как якорь (улучшение)

Для сборочных чертежей `LeaderLinePlacing.StartPoint` — произвольная точка на детали.
Центр масс детали даёт более стабильный якорь:

```csharp
var part = model.SelectModelObject(mark.ModelIdentifier) as Part;
double cogX = 0, cogY = 0, cogZ = 0;
part.GetReportProperty("COG_X", ref cogX);
part.GetReportProperty("COG_Y", ref cogY);
part.GetReportProperty("COG_Z", ref cogZ);
// Затем трансформировать (cogX, cogY, cogZ) → view coords → sheet coords
// через view.GetCoordinateSystem()
```

## Sliding anchor (улучшение для длинных деталей)

Для балок и листов якорь не точка, а отрезок (ось детали).
Оптимальный якорь — проекция текстового блока на ось детали.
Это минимизирует длину выноски при любом положении метки вдоль детали.

```
anchorOptimal = clamp(project(markCenter, partAxis), partStart, partEnd)
```
