# Роадмап расстановки видов

## Цель

Расставлять виды на чертеже так, как это делает опытный чертёжник:
с учётом проекционных связей, семантической группировки, при максимально
крупном масштабе.

Текущая реализация использует упаковщик (MaxRects) как основной механизм.
Упаковщик решает задачу **вместимости**, а не **компоновки**. Целевая
архитектура явно разделяет эти два понятия.

## Стандарты

Стандарты задают ограничения, а не алгоритм упаковки:

- ISO 128-3 — общие правила для видов, разрезов и сечений.
- ISO 128-3 делегирует расположение ортогональных проекций стандарту ISO 5456-2.
- ASME Y14.3 — создание и расположение ортогональных видов.
- Все эти стандарты определяют **граф проекционных связей**, а не задачу
  прямоугольной упаковки.

Важное уточнение:

- «Вид сверху над фронтальным» — не универсальное правило.
- Конкретное расположение (сверху/снизу/слева/справа) зависит от выбранного
  метода проецирования (первый угол / третий угол).
- Текущий роадмап опирается на соглашение, уже неявно заложенное в кодовой
  базе и в целевых чертежах.

Практический вывод:

- расстановка должна начинаться с построения графа проекций
- сначала должен быть выбран базовый проекционный вид семейства
- базовый вид выбирается для каждого чертежа отдельно
- для части building drawings плановый вид может быть одним из частых
  примеров `BaseView`, но не правилом
- упаковка подчиняется этому графу
- метод проецирования должен стать явным параметром конфигурации,
  а не скрытым допущением

## Главная идея

Расстановка видов определяется **проекционными ограничениями**, а не
эффективностью упаковки.

Для конвенции, фактически принятой в текущем модуле и целевых чертежах,
граф ограничений выглядит так:

- базовый вид (`BaseView`) — якорь проекционного семейства.
- стандартные ортогональные соседи располагаются относительно `BaseView`.
- горизонтальные разрезы тяготеют к вертикальной зоне `BaseView`
  (над/под, общий X).
- вертикальные разрезы тяготеют к боковой зоне `BaseView`
  (слева/справа, общий Y).
- конкретные имена видов (`PlanView`, `FrontView`, `TopView`, `BackView`
  и т.д.) — это частные случаи этой схемы, а не единственно возможная основа.

Упаковщик (MaxRects / shelf) — только резервный механизм для видов без
проекционной связи, либо когда проекционное размещение геометрически
невозможно на данном листе.

## Архитектурное решение

Модели `DrawingViewInfo` и объекты Tekla `View` — это транспортные
и runtime-дескрипторы. Они не являются моделью компоновки.

Целевые слои:

```
Снимок Tekla → модель проекционных связей → план зон → размещение/fallback
             → выравнивание/центрирование → DTO результата
```

Недопустимо как долгосрочный центр модуля:

- планирование через MaxRects в первую очередь
- выбор размещения разреза только по bbox
- отождествление «влезает куда угодно» с «правильно расставлено»
- смешивание проекционной семантики с ситуативными ветвями размещения

## Текущее состояние

### Публично поддерживается

- `fit_views_to_sheet`
- `get_drawing_views`
- `move_view`
- вспомогательные инструменты создания и запроса видов

### Реализовано в коде

- `DrawingViewArrangementSelector` предпочитает `FrontViewDrawingArrangeStrategy`
  при наличии `FrontView`
- `FrontViewDrawingArrangeStrategy` уже разделяет:
  - якорное/проекционно-осознанное размещение
  - fallback-упаковку для неразмещённых видов
- строгая схема резервирует явные слоты вокруг `FrontView`
- relaxed-схема резервирует место для `TopView` и основного `SectionView`
  перед размещением `FrontView`
- только наибольший `SectionView` считается основным проекционным разрезом
- остальные виды откладываются в MaxRects
- `EstimateFit` теперь проверяет неразмещённые виды в оставшемся свободном
  пространстве перед выбором масштаба
- проекционное выравнивание существует как пост-проход
- центрирование всей группы существует как финальный проход

### Подтверждённые ограничения

- проекционно-осознанное размещение есть, но только для малого фиксированного
  набора: FrontView, TopView, BottomView, BackView, один основной SectionView
- явный граф проекций в коде не строится
- роль базового вида не абстрагирована:
  - фактически якорем считается именно `FrontView`
  - альтернативный базовый вид не является first-class понятием
- это особенно ограничивает сценарии, где базовый вид не совпадает
  с `FrontView`
- обработка SectionView всё ещё area-first: один разрез выбирается как
  «основной» по размеру, остальные не классифицируются по ориентации реза
- `TryPlaceRelative` — не решатель жёстких ограничений: при неудаче
  предпочтительного размещения перебирает запасные стороны
- качество `EstimateFit` ограничено качеством модели планирования
- проекционное выравнивание и центрирование — постобработка, не ядро планировщика

Следствие:

- код уже доказывает работоспособность проекционно-осознанного планирования
- главный пробел — не «можем ли мы расставлять виды семантически»
- главный пробел — «можем ли мы явно представить и решить всю проекционную
  схему до перехода к упаковке»

## Принципы проектирования

- Проекционные ограничения — жёсткие ограничения, а не предпочтения.
- Базовый вид семейства должен быть явной сущностью, а не побочным эффектом
  поиска `FrontView`.
- Target architecture remains rule-first, optimizer-second.
- Упаковка — резервный механизм для видов без проекционной связи.
- Масштаб выбирается после того, как известна схема расстановки, не до.
- Ориентация плоскости реза определяет зону размещения, а не размер вида.
- Виды одного типа и ориентации принадлежат одной зоне.
- Расстановка должна деградировать корректно: если проекционная позиция
  не влезает — перейти к следующей лучшей, никогда не отбрасывать
  проекционную связь молча.
- Метод проецирования должен быть явным параметром, когда стандарты
  влияют на выбор стороны.
- Пост-выравнивание уточняет хороший план, а не спасает плохой.
- Центрирование и распределение пространства не должны нарушать
  проекционные связи.
- AI/ML-оптимизация допустима только поверх корректной проекционной схемы,
  а не вместо неё.

## Словарь

В этом документе термины используются в фиксированном смысле:

- `BaseView` — опорный вид конкретного чертежа, относительно которого
  строится проекционное семейство
- `BaseViewSelection` — этап выбора `BaseView` для конкретного чертежа
- `ProjectionMethod` — выбранная проекционная конвенция, определяющая
  допустимые стороны/зоны размещения зависимых видов
- `CutOrientation` — смысловая ориентация разреза: `Horizontal`,
  `Vertical`, `Unknown`
- `ProjectionLayoutPlan` — итог планирования: какие виды попали в какие зоны,
  какие деградировали, какие ушли в fallback
- `vertical zone` — зона видов, делящих с `BaseView` общую X-ось
- `side zone` — зона видов, делящих с `BaseView` общую Y-ось
- `fallback` — размещение только для тех видов, которые не удалось
  корректно разрешить в проекционном плане
- `degradation` — явное ослабление идеальной схемы компоновки с сохранением
  диагностируемости и детерминированности

## Проекционные инварианты

Планировщик должен сохранять следующие инварианты:

- `BaseView` — якорь проекционного семейства.
- стандартные проекционные соседи определяются относительно этого якоря.
- члены горизонтального проекционного семейства разделяют выравнивание по X.
- члены вертикального проекционного семейства разделяют выравнивание по Y.
- разрезы одинаковой ориентации реза принадлежат одной зоне.
- назначение зоны — сначала семантика, потом геометрия.
- fallback-упаковка никогда не переклассифицирует проекционный вид как
  свободно плавающий, если планировщик явно не зафиксировал эту деградацию.

## Целевые внутренние типы

Для реализации роадмапа достаточно зафиксировать четыре опорные сущности:

### 1. `BaseViewSelection`

Определяет, какой вид становится базовым для данного проекционного семейства.

Нужно чтобы:

- снять жёсткую привязку архитектуры к `FrontView`
- различать текущий implementation shortcut и целевую модель
- позволить planner'у работать от выбранного базового вида, а не от имени
  одного конкретного view type
- поддержать выбор `BaseView` как результата анализа конкретного чертежа

Минимальный контракт первой версии:

- `View`
- `SelectionKind`
- `Reason`
- `IsFallback`

Ожидаемая семантика:

- `View` — выбранный базовый вид или `null`, если выбор не разрешён
- `SelectionKind` — например `Resolved`, `Fallback`, `Unresolved`
- `Reason` — человекочитаемое объяснение выбора или причины неразрешённости
- `IsFallback` — быстрый признак того, что planner опирается на временное
  ослабление policy

### 2. `ProjectionMethod`

Определяет, какая стандартная схема расположения считается канонической
для текущего набора чертежей.

Нужно чтобы:

- убрать скрытое допущение про «верх/низ/лево/право»
- явно связать планировщик с выбранной проекционной конвенцией

### 3. `CutOrientation`

Фиксирует семантику разреза: `Horizontal`, `Vertical`, `Unknown`.

Нужно чтобы:

- отделить семантическую классификацию разреза от его bbox
- сделать `Unknown` явным состоянием, а не молчаливым допущением

### 4. `ProjectionLayoutPlan`

Представляет уже вычисленный план зон и размещения.

Минимально должен различать:

- якорные/проекционно запланированные виды
- деградированные, но всё ещё семантически объяснимые виды
- действительно отложенные виды, переданные в fallback-упаковку

Остальные типы (`Snapshot`, `PlacementZone`, `DeferredViewPlan` и т.д.)
могут появляться по мере реализации, но не являются целью сами по себе.

## Policy: BaseViewSelection

`BaseViewSelection` должен быть policy-driven, а не захардкоженным по имени
типа вида.

Первая версия policy должна гарантировать:

- выбор делается для каждого чертежа отдельно
- одинаковый набор входных данных даёт одинаковый результат
- при равенстве кандидатов порядок tie-break предсказуем
- unresolved case возвращается как результат выбора, а не прячется в логах

Минимальный порядок принятия решения:

1. использовать явный сильный сигнал, если он доступен
2. иначе использовать детерминированный ranking кандидатов
3. если ranking недостаточен, вернуть `Unresolved` или маркированный fallback

На уровне контракта важно:

- `PerfTrace` и diagnostics полезны, но не являются основным контрактом
- основной контракт — это `BaseViewSelectionResult`
- exception допустим только в тех calling paths, где unresolved case запрещён

## Policy: CutOrientation

Определение `CutOrientation` должно быть metadata-first.

Порядок источников:

1. Tekla metadata / coordinate system / cut direction
2. parent-view information / section mark direction
3. geometry heuristic (`aspect ratio`)
4. `Unknown`

Правила:

- более слабый источник не должен затирать более сильный
- heuristic используется только когда metadata path не дал результата
- `Unknown` остаётся валидным результатом до этапа zone assignment
- превращение `Unknown` в конкретную сторону возможно только как отдельная,
  явно записанная деградация

## Policy: Degradation

Деградация — часть целевой архитектуры, а не аварийный хвост.

Общие правила:

- деградация должна быть явной
- деградация должна быть детерминированной
- деградация должна быть объяснимой
- деградация не должна переинтерпретировать уже корректно построенные
  проекционные связи

Planner должен различать:

- `ideal` placement
- `degraded but still semantically explained` placement
- `deferred to fallback`

Минимальная диагностика деградации:

- какой вид не разрешился
- на каком шаге
- по какой причине
- был ли применён fallback

## Явные не-цели

Не строить редизайн вокруг:

- тонкой настройки оценочной функции MaxRects
- ранжирования разрезов только по площади
- aspect-ratio как основного семантического классификатора
- «лучшей эстетики» без проекционной корректности
- пост-выравнивания как замены планирования
- ML/RL как обязательного prerequisites для базового planner'а

Перечисленное остаётся полезным только как:

- резервный механизм
- эвристика
- диагностика
- тай-брейкер

## Целевая архитектура

```
BaseView (якорь)
    │
    ├── вертикальная зона (общий X):
    │      стандартные соседи "над/под" для данного projection method
    │      + горизонтальные SectionView
    │
    ├── боковая зона (общий Y):
    │      стандартные соседи "слева/справа" для данного projection method
    │      + вертикальные SectionView
    │
    └── оставшееся пространство: неклассифицированные виды → MaxRects
```

Масштаб выбирается как наибольший стандартный масштаб, при котором
полная проекционная схема влезает на лист.

Примечание:

- `BaseView` не выводится заранее из имени типа вида
- он выбирается для конкретного чертежа
- для части building drawings это может быть плановый вид
- `FrontView` для такого сценария является только одним из соседних видов,
  а не естественным якорем всей компоновки

## Внешний контекст

Внешний контекст полезен только как подтверждение направления:

- CAD-системы по-прежнему описывают drawing layout как standards-aware
  workflow, а не как чистую rectangle packing задачу.
- типичный pipeline автоматизации у вендоров строится вокруг
  `base/primary view`, стандартов, шаблонов и derived/projected views
- SOLIDWORKS 2025/2026 двигается в сторону AI-assisted generation, но даже там
  пользователь задаёт шаблон, стандарты и основной вид.
- BIM-работы 2025 с RL оптимизируют перекрытия и читаемость, но не заменяют
  явную структуру проекционных связей.

Практический вывод:

- ML/RL может стать более поздним оптимизатором читаемости, разрешения
  конфликтов или распределения пространства
- AI уместен прежде всего для локальных подзадач:
  overlap reduction, sheet selection, annotation/callout generation,
  prompt-guided refinement
- ML/RL не должен заменять явный граф проекций как основной источник
  корректности

## Фазы

### Фаза 0: BaseViewSelection

**Статус: не начата.**

Снять жёсткую привязку planner'а к `FrontView`. Сделать выбор базового вида
явной частью планирования. Зафиксировать текущий shortcut `BaseView == FrontView`
как временный fallback, а не норму. Поддержать сценарии, где `BaseView`
является произвольным видом.

Готово когда:
- planner стартует от `BaseViewSelection`, а не от прямого `FrontView` lookup
- выбор базового вида детерминирован и объясним
- fallback-выбор помечается как fallback

### Фаза 1: Классификация разрезов по ориентации плоскости реза

**Статус: не начата.**

Каждый SectionView должен нести тег `CutOrientation` (`Horizontal`, `Vertical`,
`Unknown`) до того, как planner принимает layout-решения. Истинный источник
ориентации — метаданные Tekla (`View.ViewCoordinateSystem` / направление стрелки
разреза); aspect ratio — только резервная эвристика. `Unknown` — полноправное
состояние, нельзя молча превращать в «горизонтальный».

Простая эвристика (aspect ratio):
- `width / height > порог` (например, 3.0) → горизонтальный рез → над/под
- `height / width > порог` → вертикальный рез → слева/справа
- иначе → неопределённо → остаётся как `Unknown`

Готово когда:
- каждый SectionView в planning path имеет `CutOrientation`
- planner использует этот тег при назначении вида в правильную зону

### Фаза 2: Размещение горизонтальных разрезов в вертикальной зоне BaseView

**Статус: не начата.**

Горизонтальные разрезы принадлежат вертикальной зоне базового вида:
для текущего `FrontView`-centric кода это обычно над/под `FrontView`,
но roadmap не должен зашивать именно этот частный случай в целевую
архитектуру.

Они выравниваются по оси X с `BaseView` и укладываются стопкой в этой зоне,
а не разбрасываются по листу.

Направление реализации:
- в `TryPlanRelaxedLayout` собрать все горизонтальные SectionView
- после размещения `BaseView` и стандартного соседнего вида этой зоны
  уложить горизонтальные разрезы стопкой в вертикальной зоне `BaseView`,
  отсортированных по положению плоскости реза вдоль оси, перпендикулярной
  этой плоскости (для горизонтальных разрезов стены — по Z, т.е. высотной
  отметке; ближайший к верхнему соседу —
  сверху стопки)
- зарезервировать вертикальное пространство для всей стопки до размещения
  `BaseView` (обобщение текущих `topSlotH` / `bottomSlotH`)

Готово когда:
- все горизонтальные разрезы оказываются в одной вертикальной зоне базового вида
- они выровнены по X с `BaseView`
- стопка вписывается в зарезервированный вертикальный слот

Если полный стек не влезает:

- это фиксируется как явная деградация плана
- причина деградации воспроизводима и диагностируема
- fallback не ломает уже построенные проекционные связи остальных видов

### Фаза 3: Размещение вертикальных разрезов в боковой зоне BaseView

**Статус: частично готова (только основной разрез).**

Текущий код размещает наибольший `SectionView` справа от `FrontView`
через `TryPlaceRelative(Right)`. Это текущая реализация, а не целевая
архитектура. Дополнительные вертикальные разрезы сейчас откладываются
в MaxRects.

Готово когда:
- все вертикальные SectionView укладываются стопкой в одной боковой зоне `BaseView`
- ширина этой зоны резервируется до размещения `BaseView`

Если полный стек не влезает:

- деградация фиксируется явно
- сохраняется детерминированный порядок решений
- fallback применяется только к реально неразрешённому хвосту

### Фаза 4: Выбор масштаба с учётом проекционной схемы

**Статус: частично готова.**

`EstimateFit` теперь корректно проверяет якорные + неразмещённые виды.
Однако fallback всё ещё использует MaxRects, который может отказать даже
когда проекционно-осознанная расстановка сработала бы.

Готово когда:
- `EstimateFit` использует полную проекционную схему (фазы 1–3) для
  оценки вместимости
- выбранный масштаб — наибольший, при котором влезает проекционная схема,
  а не просто какая-то упаковка

Дополнительное условие успеха:

- отказ масштаба должен быть объясним:
  - какая зона не влезла
  - какое резервирование не сработало
  - какие виды потребовали fallback

### Фаза 5: Выравнивание по X/Y после размещения

**Статус: частично готова (сервис проекционного выравнивания существует).**

После размещения видов сервис проекционного выравнивания выравнивает
стандартные соседи и `FrontView` по текущему частному сценарию. Это должно
быть обобщено до `BaseView`-centric схемы:
- все виды вертикальной зоны `BaseView` → общее выравнивание по X
- все виды боковой зоны `BaseView` → общее выравнивание по Y
- стопки разрезов одинаковой ориентации → выравнивание между собой и
  с `BaseView`

Готово когда:
- проекционное выравнивание покрывает разрезы так же, как стандартные
  проекции
- сгруппированные разрезы одинаковой ориентации выровнены между собой
  и с `BaseView` по соответствующей оси

### Фаза 6: Центрирование и распределение пространства

**Статус: базовое центрирование существует.**

После проекционного размещения группа видов центрируется на свободной
площади листа. Сейчас есть простой проход центрирования.

Улучшение: распределять пространство между зонами (выше/ниже/слева/справа)
пропорционально, а не концентрировать его на одной стороне.

## Ближайший checklist

### Phase 0: BaseViewSelection

Цель:

- убрать жёсткую привязку planner'а к `FrontView`
- сделать выбор базового вида явной частью планирования

Шаги:

- ввести внутреннюю модель `BaseViewSelectionResult`
- определить стратегию выбора базового вида для одного чертежа
- вернуть из выбора базовый вид + объяснение причины выбора
- заменить прямой поиск `FrontView` в planning path на результат `BaseViewSelection`
- сохранить текущий `FrontView`-based shortcut как временный fallback, а не как норму

Минимальные правила первой версии:

- если в чертеже есть явно подходящий базовый вид, planner работает от него
- если надёжно определить базовый вид нельзя, это фиксируется явно
- временный fallback допускается, но должен быть диагностируемым

Тесты:

- выбирается базовый вид без жёсткой зависимости от `FrontView`
- одинаковый набор входных видов даёт одинаковый выбор `BaseView`
- fallback-выбор помечается как fallback

Готово когда:

- planner стартует от `BaseViewSelection`, а не от прямого `FrontView` lookup
- выбор базового вида детерминирован и объясним

### Phase 1: CutOrientation

Цель:

- классифицировать `SectionView` по смысловой ориентации реза до layout-решений

Шаги:

- ввести `CutOrientation`
- собрать доступные Tekla-метаданные, пригодные для определения ориентации
- реализовать определение ориентации по метаданным как основной путь
- реализовать aspect-ratio heuristic как fallback
- сделать `Unknown` явным результатом, а не скрытым допущением
- пробросить `CutOrientation` в planning path

Минимальные правила первой версии:

- metadata-first
- heuristic-second
- `Unknown` сохраняется до этапа принятия layout-решения

Тесты:

- горизонтальный разрез определяется как `Horizontal`
- вертикальный разрез определяется как `Vertical`
- ambiguous case не схлопывается молча
- отсутствие метаданных переводит классификацию в fallback path

Готово когда:

- каждый `SectionView` в planning path имеет `CutOrientation`
- planner использует `CutOrientation` при назначении зоны

## Стратегия валидации

Роадмап должен выполняться с тестами трёх уровней:

### 1. Юнит-тесты планировщика

- классификация ориентации разреза
- резервирование зон до размещения якоря
- детерминированный порядок стопки внутри зоны
- решения о деградации когда предпочтительное размещение не влезает

### 2. Геометрические тесты / тесты компоновки

- нет перекрытий между запланированными видами
- нет перекрытий с зарезервированными зонами (штамп, спецификация и т.д.)
- инварианты выравнивания по X/Y для проекционных семейств
- стабильный результат при повторном `fit_views_to_sheet`

### 3. Live-валидация в Tekla

- проверить, что runtime-координаты и смещения фреймов Tekla сохраняют
  запланированную топологию после `Modify()` / `CommitChanges()`
- проверить, что метаданные разреза для определения ориентации действительно
  доступны и стабильны на живых чертежах
- проверить взаимодействие зарезервированных зон со штампами, таблицами
  ревизий и спецификациями на реальных листах

## Критерии приёмки

Расстановка корректна в штатном случае когда:

- `BaseView` находится в визуальном центре чертежа, не прижат к углу
- стандартные проекционные соседи расположены корректно относительно `BaseView`
  в соответствии с выбранным `ProjectionMethod`
- все горизонтальные SectionView находятся в одной вертикальной зоне `BaseView`,
  выровнены по X с `BaseView`, сгруппированы вместе
- все вертикальные SectionView находятся в одной боковой зоне `BaseView`,
  выровнены по Y с `BaseView`, сгруппированы вместе
- выбранный масштаб — наибольший, при котором эта схема влезает
- нет перекрытий с зарезервированными зонами
- результат стабилен: двойной запуск `fit_views_to_sheet` даёт одну и ту
  же компоновку

Расстановка корректна в деградированном случае когда:

- планировщик явно сообщает, какие виды не удалось удержать в канонической зоне
- причина деградации объяснима и повторяема
- уже размещённые проекционные связи не нарушаются ради упаковки остатка
- fallback затрагивает только неразрешённые виды, а не переинтерпретирует
  весь layout

## Сигнал завершения роадмапа

Редизайн на верном пути когда:

- планировщик может объяснить каждый размещённый вид в семантических терминах:
  - якорь
  - горизонтальная проекционная зона
  - вертикальная проекционная зона
  - деградированный fallback
- размещение SectionView больше не определяется главным образом ранжированием
  по размеру
- `EstimateFit` и итоговая расстановка опираются на одну и ту же
  проекционную модель
- пост-выравнивание только уточняет, а не переинтерпретирует компоновку

Редизайн готов к дальнейшей оптимизации когда:

- граф проекций явно представлен в коде
- планирование зон детерминировано
- пути деградации явны и тестируемы
- упаковчик затрагивает только действительно незапланированные
  или семантически свободные виды
